  1. Smoothing Onboarding to Open Source Software Development. Is it possible?

    2. [Dec, 16 , 2022 - Emad Aghaei]
  2. There are several barriers that can make onboarding to open source software development challenging:
    1. Lack of knowledge or experience: If you are new to software development or open source, it can be intimidating to get started. You may not know where to find resources or how to contribute effectively.
    2. Complex codebases: Some open source projects have large, complex codebases that can be overwhelming for newcomers to navigate.
    3. Lack of clear documentation: If a project lacks clear documentation, it can be difficult for newcomers to understand how it works and how they can contribute.
    4. Lack of community support: Some open source projects may not have active communities or may not be welcoming to newcomers. This can make it difficult to get help or guidance when needed.
    5. Difficult contribution process: Some open source projects may have a complex or rigid contribution process, which can be intimidating or confusing for newcomers.
    6. Lack of time: Open source development can be time-consuming, and it may be difficult for newcomers to find the time to get involved.
    3. There are a few ways to remove or reduce onboarding challenges in open source software development:
    1. Provide clear and detailed documentation: Make sure that there is clear and detailed documentation available for newcomers to refer to. This can include guides on how to get started, how to contribute, and how the codebase is structured.
    2. Encourage a welcoming and supportive community: Create a welcoming and supportive community culture that is open to helping newcomers get started and get involved. This can include providing resources and guidance, as well as being patient and helpful when answering questions.
    3. Make the contribution process clear and simple: Make sure that the contribution process is clear and simple, with guidelines and resources available to help newcomers understand how to contribute effectively.
    4. Offer opportunities for newcomers to get involved: Offer opportunities for newcomers to get involved with small, self-contained tasks or issues to help them get started and build up their skills.
    5. Provide mentorship and guidance: Consider offering mentorship or guidance to newcomers to help them get up to speed and become more comfortable with the codebase and community.

    Microtask Programming as a solution

    We believe adapting microtask programming can address several challenges exists in open source software develpment. Microtask programming refers to the practice of breaking down larger programming tasks into smaller, more manageable pieces that can be completed independently. This approach can have a number of impacts on programming, both positive and negative:

    Positive impacts:
    1. Microtask programming can make it easier for developers to focus on one small task at a time, which can help reduce cognitive load and improve productivity.
    2. It can also make it easier for developers to collaborate on larger projects, as each developer can work on their own microtask without having to coordinate with others.
    3. Microtask programming can also be useful for distributed teams, as it allows developers to work on their own tasks independently and at their own pace.
    Negative impacts:
    1. Microtask programming can sometimes lead to a lack of overall context, as developers may not have a full understanding of the larger project or how their microtask fits into the overall picture.
    2. It can also lead to a lack of ownership or responsibility, as each developer is only working on a small part of the project and may not feel as invested in the overall outcome.
    3. Microtask programming may also require more overhead and coordination, as each microtask must be clearly defined and tracked separately.

    To investigate the potential benefits and drawbacks of microtask programming we conducted a controlled experiment. We focused our investigation on the context in which microtasking is most widely used, implementing and debugging function bodies, and investigated the impact of microtasking with respect to onboarding, project velocity, code quality, and developer productivity. 28 developers worked to implement microservice endpoints, either in the form of traditional programming tasks described in an issue tracker or as programming microtasks. Our study did not examine the design effort necessary to prepare for microtask programming or consider how microtask programming might be applied to maintenance tasks. We found that, compared to traditional programming, microtask programming reduced onboarding time by a factor of 3.7, increased project velocity by a factor of 5.76 by increasing team size by a factor of 7, and decreased individual developer productivity by a factor of 1.3. The quality of code did not significantly differ. Through qualitative analysis of how developers worked, we explore potential explanations of these differences. These findings offer evidence for the potential benefits that adopting microtask programming might bring, particularly in cases where increasing project velocity is paramount.
